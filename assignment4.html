Introduction

This website will give a thorough evaluation of the website we created for Webtech assignments, using the Web Content Accessibility Guidelines (WCAG) 2.0 and feedback from Teaching Assistants.

Appendix A will assess the accessibility of the page according to the WCAG's success levels (A, AA, AAA), using provided requirements. For each guideline, a short discussion and conclusion will be included for clarification or making comparisons where relevant.

Appendix B will compile and categorize TA feedback.

Overall, our website comes close to meeting Accessibility Level A. The website meets many of the more important requirements listed by the WCAG, and the impact of unfulfilled requirements is not high due to the limited amount of content (text, links, images, etc.) on our page.

With some improvements, which will be discussed in the Improvement Plan section, the website can reach success level A.

This documentation uses a CSS file "stylesheetassignment4.css" to create table borders for it to be more readable.

Accessibility Evaluation

This section will provide an evaluation of our website according to the WCAG, whether it meets level A or not, where relevant & appropriate to our website's contents, as the webpage is very basic and minimal.

Appendix A: Web Content Accessibility Guidelines (WCAG)
Guideline Level Fulfilled Requirements Missing Requirements Evaluation
Guideline 1.1 Text Alternatives 1.1.1: Non-text Content A
  • FLASH1: Setting the name property for a non-text object
  • C18: Using CSS margin and padding rules instead of spacer images for layout design
  • FLASH27: Providing button labels that describe the purpose of a button
  • FLASH25: Labeling a form control by setting its accessible name
  • H37: Using alt attributes on img elements

The website meets success level A.

  • With the exception of the side navigation bar, each of our buttons had names that accurately describes its purpose.
  • As for the sidebar, the functionality of the buttons is not explicitly stated.
  • We did not include an alt attribute on our map. This can be added to the source code in our improvements phase.
  • Our reset button did not explicitly state how users can use it, this can be addressed with some minor changes in HTML.
Guideline 1.3 Adaptable 1.3.1: Info and Relationships A
  • ARIA2: Identifying a required field with the aria-required property fulfilled by having "required" HTML tags
  • C22: Using CSS to control visual presentation of text
  • G117: Using text to convey information that is conveyed by variations in presentation of text
  • G141: Organizing a page using headings
  • H39: Using caption elements to associate data table captions with data tables
  • H42: Using h1-h6 to identify headings
  • H43: Using id and headers attributes to associate data cells with header cells in data tables
  • H48: Using ol, ul and dl for lists or groups of links
  • H49: Using semantic markup to mark emphasized or special text
  • H51: Using table markup to present tabular information
  • H97: Grouping related links using the nav element
  • FLASH29: Setting the label property for form components
  • H44: Using label elements to associate text labels with form controls
  • H65: Using the title attribute to identify form controls when the label element cannot be used

The website does NOT meet success level A.

  • We initially found that including a title/label for the form was not necessary since the table header already gave info about what the user needed to input.
  • Combine this with the placeholders for the input fields, users should have a general idea of what they need to input.
  • However, this can cause problems when the table has a large number of items, as users may have to scroll up to check the headings / or have already filled out some fields wrong (as the placeholders will disappear).
  • We will discuss how to address this problem in the Improvements section.
1.3.2: Meaningful Sequence A
  • C27: Making the DOM order match the visual order
  • G57: Ordering the content in a meaningful sequence
  • PDF3: Ensuring correct tab and reading order in PDF documents

The website meets success level A.

  • Because we came up with ideas before starting to code, we had time to organize our DOM and content's order in a way that makes sense.
  • The reading order is not affected when converting the webpage to PDF.
Guideline 1.4 Distinguishable 1.4.1 & 1.4.3: Use of Color & Contrast A
  • C15: Using CSS to change the presentation of a user interface component when it receives focus
  • G14: Ensuring that information conveyed by color differences is also available in text
  • G183: Using a contrast ratio of 3:1
  • G174: Providing a control with a sufficient contrast ratio that allows users to switch to a presentation that uses sufficient contrast
  • G205: Including a text cue for colored form control labels

The website does NOT meet success level A.

  • Our main failure comes from failing to include a text cue to inform users about required form fields, although this can be improved with some minor changes to the source code.
  • We mostly used white text on a black/dark gray background and the other way around, ensuring a large contrast between text and background for a better reading experience.
1.4.4: Resize Text AA
  • C12: Using percent for font sizes
  • C20: Using relative measurements to set column widths
  • C22: Using CSS to control visual presentation of text
  • SCR34: Calculating size and position in a way that scales with text size
  • G179: Ensuring that there is no loss of content or functionality when the text resizes and text containers do not change their width

The website fulfills the basic CSS requirements for this guideline.

  • We used percentages for most of our containers (columns, table, navbar) and font sizes, as it is relative and easier to scale when the viewport dimensions change.
  • However, this can cause problems with resizing widths, and there is loss of content of the table or navbar when the viewport is too narrow.
1.4.8: Visual Presentation AAA
  • C19: Specifying alignment either to the left OR right in CSS
  • C21: Specifying line spacing in CSS
  • C23: Specifying text and background colors of secondary content such as banners, features and navigation in CSS while not specifying text and background colors of the main content
  • C24: Using percentage values in CSS for container sizes
  • G188: Providing a button on the page to increase line spaces and paragraph spaces
  • G204: Not interfering with the user agent's reflow of text as the viewing window is narrowed
  • G206: Providing options within the content to switch to a layout that does not require the user to scroll horizontally to read a line of text

The website fulfills the necessary CSS requirements.

  • We have managed to separate CSS (such as background-color or text-color) from HTML and avoided (mostly) using styles in HTML code.
  • CSS alignment is used in the cellphones table, and line spacing is used in the sidebar for cleaner separation between buttons.
  • The unfulfilled requirements were difficult for us to implement at the time of assignment 1, although it is possible with some extra time.
Guideline 2.1 Keyboard Accessible 2.1.1: Keyboard A
  • H91: Using HTML form controls and links
  • G90: Providing keyboard-triggered event handlers
  • G202: Ensuring keyboard control for all functionality

The website meets success level A.

  • Our website has very few functionalities that can be used with a keyboard.
  • Simple actions can be done via keyboard: Submit form (Enter), PgUp & PgDn
  • However, resetting the database is not possible with only a keyboard.
2.1.2: No Keyboard Trap A
  • G21: Ensuring that users are not trapped in content

The website meets success level A.
Guideline 2.4 Navigable 2.4.1: Bypass Blocks A
  • G1: Adding a link at the top of each page that goes directly to the main content area
  • G124: Adding links at the top of the page to each area of the content
  • H69: Providing heading elements at the beginning of each section of content
  • G123: Adding a link at the beginning of a block of repeated content to go to the end of the block

The website meets success level A.

  • The sidebar provides the functionalities needed for requirements G1 & G123 & G124.
  • In our opinion, requirement G123 is not necessary for a single-page website with little content like ours.
  • Users can navigate to each section of the page using the anchors in the navbar, as well as go to the main content area (cellphone table).
  • Since we only had 2 sections, remembering to provide headings for each was not difficult.
2.4.4: Link Purpose (In Context) A
  • G91: Providing link text that describes the purpose of a link
  • H30: Providing link text that describes the purpose of a link for anchor elements
  • FLASH27: Providing button labels that describe the purpose of a button
  • H24: Providing text alternatives for the area elements of image maps

The website does NOT meet success level A.

  • Our links and anchors have clear and easy-to-remember names.
  • Each of our buttons also have a clear label that describes its purpose.
  • Requirement H24 can be implemented with some small changes to the HTML source code.
2.4.6: Headings and Labels AA
  • G131: Providing descriptive labels partially fulfilled

The website meets success level AA.

  • Requirement G130: "Providing descriptive headings" was not included due to the limited amount of content (and headings) on the page. If we had included an extra "Accessories" section for our website, a descriptive heading would have been necessary.
  • Although there are no labels, the submit form has placeholders that gives information on how to fill out the form. However, we did not show clearly which fields are required and which fields are not.
2.4.8: Location AAA
  • G128: Indicating current location within navigation bars

For this guideline, we have tried to use the CSS :active selector to highlight where the user is on the page, but failed to get it working.
